## Sprint Overview
Sprint Number: [N]
Theme: [Name]

## Planning Meeting (2h, Monday)
Attendees: Pod Lead, PM, EM, QA Lead, Senior Engineers

## Step 1: Review Previous Sprint
- Story points committed vs delivered
- Defects found, quality gate results
- Retro actions carried forward

## Step 2: Define Sprint Goal
- Goal: [One sentence]
- Success criteria: [3 measurable]
- Non-goals: [What is OUT of scope]

## Step 3: Capacity Planning
- Never allocate > 80% of available capacity
- Reserve 20% for: unexpected issues, code review, tech debt

## Step 4: Sprint Backlog
### Must Have (P0, committed)
| ID | Feature | Owner | Points | Dependencies |
### Should Have (P1, best effort)
### Could Have (P2, if capacity allows)

## Step 5: Dependency Map
- Cross-pod dependencies with status and risk
- Critical deps resolved in week 1

## Step 6: Quality Gates
- Code review: all PRs approved
- Unit tests: > 80% coverage
- Integration tests: critical paths
- Security scan: 0 critical/high
- Accessibility: WCAG 2.1 AA
- Performance: within budget
- Documentation: ADRs + API docs

## Step 7: Risk Register
| Risk | Probability | Impact | Mitigation | Owner |

## Step 8: Communication Plan
- Daily pod sync
- Weekly status to stakeholders
- Twice-weekly dependency review
- Fortnightly portfolio review

## Execution Checklist
### Week 1: Planning, first code reviews, mid-sprint check-in
### Week 2: Features complete, testing, quality gates, release prep, demo, retro

## Sprint Review (1h, Friday)
- Goal recap (2 min)
- Feature demos (30 min)
- Metrics (10 min)
- Stakeholder feedback (15 min)

## Sprint Retrospective (1h, Friday)
- What went well (15 min)
- What didn't (15 min)
- Root cause analysis (15 min)
- Action items (15 min)
